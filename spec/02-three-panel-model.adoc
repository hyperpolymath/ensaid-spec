== Abstract

This section defines the three-panel model that structures the visual and logical layout of an eNSAID-compliant environment. The three panels -- Panel-L (Symbolic), Panel-N (Neural), and Panel-W (World) -- map directly to the Binary Star architecture's three gravitational masses.

== Motivation

A neurosymbolic development environment must simultaneously present three distinct information streams: the symbolic constraints that govern behaviour, the neural reasoning that generates output, and the shared workspace where validated results appear. Traditional IDE layouts (editor + terminal + sidebar) do not accommodate this tripartite information flow. DAW-style parallel views (multiple simultaneous channels) are a closer analogy.

== Specification

=== S02-1: Panel Definitions

A conforming eNSAID environment MUST provide three primary panels:

==== Panel-L (Symbolic Mass)

Panel-L represents the Human Star's symbolic constraints. It MUST support:

* Displaying active constraints that govern Machine behaviour.
* Adding, editing, and removing constraints interactively.
* Marking constraints as pinned (persistent across sessions) or transient.
* Toggling constraints between active and inactive states without deleting them.

Constraint types that a conforming implementation SHOULD support:

[cols="1,3,2"]
|===
| Type | Description | Enforcement Point

| Type constraint
| Output must conform to a structural type (e.g., "valid JSON").
| Compile-time or runtime validation.

| Safety constraint
| Behavioural restriction (e.g., "no filesystem writes without confirmation").
| Runtime validation in Anti-Crash Gate.

| Domain constraint
| Domain-specific quality rule (e.g., "generated code must pass linter").
| Post-generation validation.

| Logical constraint
| Formal logical relationship (e.g., "if A then B").
| Formal verification backend (if available).
|===

==== Panel-N (Neural Stream)

Panel-N represents the Machine Star's neural reasoning stream. It MUST support:

* Displaying a stream of tokens generated by the Machine Agent, each with a timestamp and confidence score.
* Showing the Machine's internal reasoning or monologue when available.
* Indicating the current OODA phase (or equivalent reasoning phase) of the Machine Agent.
* Displaying the validation status of each token (validated, rejected, pending).

==== Panel-W (World / Barycentre)

Panel-W represents the shared workspace -- the Barycentre around which both stars orbit. It MUST support:

* Displaying only validated output (tokens that have passed the Anti-Crash Gate).
* Serving as the canonical "ground truth" of the current task state.
* Supporting both Human and Machine modifications, subject to governance rules.

=== S02-2: Panel Layout

Panels SHOULD be arranged in a parallel side-by-side layout emphasising simultaneity:

* Panel-L on the left (Symbolic / Human).
* Panel-N in the centre (Neural / Machine).
* Panel-W on the right (World / Barycentre).

A conforming implementation MAY support alternative layouts (stacked, tabbed, floating) provided all three panels remain accessible and their distinct roles are preserved.

=== S02-3: Panel Toggle

A conforming environment MUST allow the operator to show or hide individual panels without destroying their state. The implementation SHOULD provide keyboard shortcuts for toggling panels. Suggested bindings:

* `Ctrl+Shift+L` -- Toggle Panel-L
* `Ctrl+Shift+N` -- Toggle Panel-N
* `Ctrl+Shift+W` -- Toggle Panel-W

=== S02-4: Additional Panels

The three primary panels (L, N, W) define the minimum viable eNSAID layout. A conforming environment MAY provide additional panels beyond the core three. Additional panels MUST:

* Declare which primary panel they relate to (L, N, W, or cross-cutting).
* Follow the panel contract format defined in <<eNSAID Section 05: Panel Contract Format>>.
* Be classifiable within the panel taxonomy defined in <<eNSAID Section 06: Panel Taxonomy>>.

=== S02-5: Data Flow Between Panels

The canonical data flow between the three primary panels is:

----
Panel-L (constraints) ---> Anti-Crash Gate <--- Panel-N (tokens)
                                |
                                v
                           Panel-W (validated output)
----

* Panel-N generates tokens.
* The Anti-Crash Gate (see Section 03) validates each token against Panel-L constraints.
* Validated tokens flow to Panel-W.
* Rejected tokens trigger operator intervention.

A conforming implementation MUST NOT allow unvalidated Machine tokens to appear in Panel-W without passing through some form of constraint validation.

=== S02-6: Dark Start Mode

A conforming environment MAY implement Dark Start Mode, where the environment launches with Panel-W showing a topology diagram or welcome state and Panel-L and Panel-N in an empty/inactive state. Activation (first keypress or mouse click) transitions to the active co-orbit state. This reduces startup anxiety and emphasises intentionality.

== Conformance

An eNSAID implementation conforms to this section if it:

1. Provides the three primary panels (L, N, W) with the specified roles.
2. Ensures unvalidated Machine output does not reach Panel-W.
3. Allows toggling panels without state destruction.
4. Supports adding constraints interactively in Panel-L.

== Reference Implementation

PanLL implements the three-panel model with:

* `PaneL.res` -- Symbolic constraints panel with interactive editor.
* `PaneN.res` -- Neural stream panel with token display and OODA phase indicator.
* `PaneW.res` -- World panel for validated workspace output.
* Keyboard shortcuts: `Ctrl+Shift+L`, `Ctrl+Shift+N`, `Ctrl+Shift+B` (Panel-W).
* Dark Start Mode with Binary Star topology diagram on initial launch.
